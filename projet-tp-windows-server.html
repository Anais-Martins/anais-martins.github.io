Administration d'un environnement Windows Server

Contexte

Date : Mai – Juillet 2025

Cadre : École – TP administration système

Ce TP s'inscrit dans la continuité du TP Réseaux de l'entreprise fictive AssurTech. L'objectif était de mettre en place un environnement d'entreprise complet à l'aide de Windows Server 2019 et Windows 10 dans un laboratoire virtualisé sous VirtualBox, en simulant une infrastructure multi-sites réaliste.

Objectif

Déployer et administrer un environnement Windows Server complet pour la société AssurTech, comprenant la mise en place d'un Active Directory, la gestion des comptes utilisateurs, la configuration des services réseau (DNS, DHCP), la gestion des dossiers partagés avec permissions NTFS, et la définition de stratégies de groupe (GPO).

Environnement virtuel mis en place

L'environnement repose sur deux machines virtuelles interconnectées via un réseau NAT nommé winsrv, déployées sur VirtualBox.

🖥 WS-SRV-DC01

OS : Windows Server 2019 Standard Evaluation

  • Contrôleur de domaine (ADDS)
  • Serveur DNS
  • Serveur DHCP
  • IP statique : 192.168.200.4
  • Domaine : assurtech.local

💻 win_Client

OS : Windows 10 Pro

  • Poste client du domaine
  • Configuration réseau via DHCP
  • Utilisé pour tester les connexions, les droits d'accès et les GPO

Mon rôle

J'ai réalisé l'intégralité de ce TP seule, de la configuration initiale du serveur jusqu'au déploiement des GPO. J'ai installé et paramétré chaque rôle Windows Server, créé l'ensemble de la structure Active Directory (OUs, groupes, utilisateurs), géré les partages réseau avec les permissions NTFS, et configuré les stratégies de groupe. J'ai également rédigé le rapport PDF illustré de captures d'écran couvrant toutes les étapes.

Étapes de configuration

1

Configuration de l'adresse IP statique du serveur

Depuis "Local Server", désactivation d'IPv6 et saisie manuelle des paramètres IPv4 (192.168.200.4 / masque 255.255.255.0 / passerelle 192.168.200.1). DNS secondaire : 8.8.8.8. Renommage de la machine en WS-SRV-DC01.

2

Installation des rôles ADDS et DNS

Via le Gestionnaire de serveur > "Ajouter des rôles et fonctionnalités", installation des rôles Active Directory Domain Services (ADDS) et DNS. Promotion du serveur en contrôleur de domaine via l'assistant de configuration post-déploiement, création d'une nouvelle forêt avec le domaine assurtech.local (NetBIOS : ASSURTECH). Mot de passe DSRM : RobustPassword123!

3

Configuration du DNS

Création d'une zone de recherche inversée IPv4 (200.168.192.in-addr.arpa) depuis le Gestionnaire DNS. Activation des mises à jour dynamiques sécurisées (intégrées à Active Directory). Vérification du bon fonctionnement via la commande nslookup : résolution de assurtech.local vers 192.168.200.4 confirmée.

4

Installation et configuration du DHCP

Installation du rôle Serveur DHCP, autorisation dans Active Directory avec le compte ASSURTECH\Administrateur. Création de l'étendue Plage-Clients :

Plage distribuée192.168.200.10 → 192.168.200.60
Plage exclue192.168.200.40 → 192.168.200.60
Passerelle par défaut192.168.200.1
Serveur DNS192.168.200.4
Domaine parentassurtech.local
Réservation serveur AD192.168.200.4
5

Jonction du client Windows 10 au domaine

Configuration du client en DHCP (ipconfig /release → /renew → /all pour vérification). L'adresse 192.168.200.10 a été attribuée automatiquement. Jonction au domaine assurtech.local via Propriétés système > Modifier, authentification avec le compte administrateur du domaine. Vérification par nslookup assurtech.local depuis le client.

6

Création de la structure Active Directory

Via "Utilisateurs et ordinateurs Active Directory" : création des 4 unités d'organisation (OU), des groupes de sécurité par service, et des utilisateurs fictifs (2 par groupe minimum), puis ajout de chaque utilisateur dans son groupe correspondant.

7

Création des dossiers partagés et permissions NTFS

Via le Gestionnaire de serveur > Services de fichiers > Partages : création d'un dossier partagé par service (profil SMB-Rapide). Désactivation de l'héritage des permissions, suppression des entrées génériques, puis ajout du groupe de sécurité correspondant avec les droits Modification sur le dossier.

8

Configuration des GPO

Création de deux GPO distinctes via la "Gestion de stratégie de groupe" : une GPO de déploiement logiciel (liée à chaque OU) et une GPO de restriction CMD (liée à toutes les OU sauf Technique-MRS via filtrage de sécurité).

Dossiers partagés et permissions

Un dossier partagé par service a été créé sur le serveur, accessible via le chemin UNC \\WS-SRV-DC01\[Nom-share]. Les permissions NTFS ont été configurées afin que seul le groupe correspondant puisse accéder à son dossier.

Stratégies de groupe (GPO)

📦 GPO – Déploiement logiciel

Liée à chaque OU (TLS, PAR, MRS, BDX). Installation automatique au démarrage machine des logiciels suivants (fichiers .msi stockés dans le partage Technique-share) :

  • Mozilla Firefox 140.0.2
  • 7-Zip 24.09 (x64)
  • Microsoft Teams (Machine-Wide)

🚫 GPO – Restriction CMD

Liée à toutes les OU du domaine. Désactive l'accès à l'invite de commande (cmd.exe) pour tous les utilisateurs, à l'exception du groupe Technique-MRS qui bénéficie d'un filtrage de sécurité en refus d'application.

  • Paramètre : Configuration utilisateur > Système > Désactiver l'accès à l'invite de commandes
  • Valeur : Activé
  • Exception : groupe Technique-MRS exclu via délégation

Problèmes rencontrés et solutions apportées

  • Lenteur des machines virtuelles : les VMs étaient parfois très lentes, rendant certaines manipulations longues et risquées (risque d'erreur). Solution : patience, redémarrage régulier et sauvegarde d'instantanés VirtualBox avant les opérations critiques.
  • Mauvaise configuration d'un dossier partagé : un paramètre de permission NTFS avait été mal défini (principal mis sur un utilisateur plutôt que sur le groupe). Solution : suppression du partage et recréation complète avec le bon groupe comme principal.
  • Jonction du client au domaine : lors de la première tentative, une erreur DNS (RCODE_NAME_ERROR) empêchait de contacter le contrôleur de domaine. Cause : le client utilisait encore 8.8.8.8 comme DNS préféré. Solution : configuration du DNS préféré du client sur l'adresse du serveur (192.168.200.4).
  • Fichier .msi de Microsoft Teams introuvable : le package MSI de Teams n'était pas disponible directement sur le site officiel. Solution : recherche sur plusieurs sources pour récupérer le fichier Teams Machine-Wide Installer compatible GPO.
  • Filtrage GPO CMD pour le service Technique : difficulté à trouver l'option "Appliquer la stratégie de groupe" dans les permissions de délégation de la GPO liée à l'OU MRS. La GPO a été appliquée à toutes les OU et le groupe Technique-MRS exclu via les paramètres de sécurité avancés.
